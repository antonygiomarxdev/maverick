## Plan: Fix Cross-Compiled SPI Binary Initialization

### Goal
Fix the cross-compiled aarch64 release binary so `lgw_start()` succeeds on Raspberry Pi + RAK2287, and add CI verification to prevent regression.

### Context
**UPDATED:** Cross-compiled binary fails at `lgw_start` with `ERROR: Failed to set SX1250_0 in STANDBY_RC mode`. Systematic debugging disproved the original struct layout hypothesis. The actual root cause is that our code never performed the **GPIO hardware reset** required by sx1302_hal before `lgw_start()`. The Semtech packet forwarder runs `./reset_lgw.sh start` before `lgw_start()`; without it, the SX1302/SX1250 may be in a bad state from previous runs, causing SPI communication with the radios to fail.

<action>
If Task 2 reveals layout differences:

1. **Option A - Pack the struct:**
   ```c
   struct __attribute__((packed)) lgw_conf_rxrf_s {
       // ... fields
   };
   ```
   This forces consistent layout across compilers. Risk: may affect performance on some architectures.

2. **Option B - Add explicit padding fields:**
   ```c
   struct lgw_conf_rxrf_s {
       uint32_t freq_hz;
       float rssi_offset;
       uint8_t type;
       uint8_t tx_enable;
       uint8_t _padding[2]; // Explicit padding for 4-byte alignment
   };
   ```

3. **Option C - Use compiler flags to match padding:**
   Add `-fpack-struct` or `-mstructure-size-boundary=8` to build.rs for cross-compilation.

4. **Recommended:** Start with Option B (explicit padding) as it's the safest and most portable.

If layout is identical, investigate SPI speed/timing:
- Check `SPI_IOC_MESSAGE` ioctl speed settings in libloragw
- Verify `config.h` `SPI_SPEED_HZ` is identical between builds
- Add delays after chip reset if needed
</action>
